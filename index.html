관심있는 개발 분야

가장 관심 있는 분야는 프론트 뷰 개발입니다.

다양한 프로젝트(구축 2건, 개선/유지보수 4건, 운영 1건)를 통해, HTML로 의미 구조를 설계하고, 자바스크립트가 사용하기 쉬운 마크업을 작성했습니다. 디자인 가이드 맞게 컴포넌트 마크업 작업을 진행했습니다.

  • Vue프로젝트를 통해, 컴포넌트 마크업의 중요성을 배움

    1년가량의 삼성생명 VueJs 프로젝트에서, 공통 컴포넌트 설계를 위해 프론트 개발자와 협업한 적이 있습니다. 퍼블리셔가 작성한 마크업을, 프론트 개발자가 Vue컴포넌트로 변경하고, 완성된 부품을 페이지로 조립하는 업무였습니다. 전달받은 컴포넌트에서 HTML 태그 속성이 빠져있거나, 추가되는 마크업 구조 때문에, 수정/요청 작업이 많았습니다. 산출물이 완성되기까지 페이지 작업이 잠시 보류되기도 했습니다. 좋은 컴포넌트는 마크업이 제대로 작성해야 함을 배웠습니다.

  • 3개월 JQuery 토이 프로젝트를 통해, 컴포넌트 단위 개발 장점을 이해

    실무에서 정적(Static) 페이지를 구축하면서, HTML을 동적으로 다루는 스크립트 역량에 관심 두게 됐습니다. 3개월 동안 JQuery 토이 프로젝트를 만들며, DOM 제어하는 스크립트 작성과 프로젝트 구조설계를 경험했습니다. JS를 로드(load) 하는데 매번 마크업 파일에 스크립트를 적용하고, 같은 HTML을 반복해 적는 것이 당연했지만 개선하고 싶은 부분이었습니다. 이후 리액트를 학습하며, 당시 겪었던 불편함이 현재 컴포넌트 개발방식이 태생하게 된 이유임을 깨달았습니다.

  • 3개월 React 토이 프로젝트를 통해, 유지보수 관점의 코드설계 중요성을 이해

    • 데이터 CRUD를 연습하고, 리액트 학습 목적으로, 중고마켓을 주제로 웹 서비스를 구축했습니다. 개발스택은 입문자들이 웹 서비스를 구현하기 쉬운 MERN(MongoDB, ExpressJs, ReactJs, NodeJs)으로 선택했고, DB 설계와 배포까지 웹 개발 프로세스를 경험했습니다.

    • 개발 유튜브 영상을 통해 프로젝트 세팅을 이해하고, 리액트-노드 작동 방식을 벤치마킹했습니다. 웹 서비스 배포 이후, 거듭된 코드 리팩토링하면서, 핵심 로직을 정의하고 관심사 분리로 컴포넌트 설계하는 중요성을 익혔습니다. 토이 프로젝트를 코드리뷰 받으면서, 제품이 서비스로 완성되는 것만큼, 유지보수하기 좋은 설계와 코드 안정성/신뢰성이 중요하다는 인사이트를 얻었습니다.

개발에 관심 갖게 된 계기와 노력

  • 개발에 관심 갖게 된 계기

    '이거 내가 수정 할 수 있을까?'

    • 주말 평균 100명가량의 수강생이 찾는 학원에서, 회계 솔루션 제품을 사용하며, 수납 업무 일한 적 있습니다. 학기 초나 신규강좌 준비기간이면, 강의 등록을 위해 수강생들로 인산인해였습니다.

    • 일할 때 어려웠던 점은, 사람마다 사정이 다양하게 있어 프로그램에 정의된 기능으로 처리 불가한 사례가 종종 발생했습니다. 이런 경우 해당 내용을 메모해 두는 것으로 문제 해결했습니다. 사용 중인 솔루션 콤보박스(Select)에 option 하나 더 추가 할 수 있었다면, 굳이 수작업이 필요 없었습니다.

    • 불편한 점을 회사에 건의하려 했지만, 매번 경우마다 외주회사에 솔루션 수정요청할 수 없었고, 작업자 사이에서 메모 작성을 가능한 업무로 생각했기 때문에, 더 이상 문제를 개선할 수 없었습니다.

  • 개발자로 성장하기 위해 어떤 노력

    • 웹 퍼블리셔, 논리적 구조로 HTML 작성

      취업 성공 패키지를 수료 후, 첫 개발회사에서 마크업 업무를 맡았습니다. 홈페이지의 일부 마크업 수정을 시작으로, 홈페이지 리뉴얼, 이메일 뉴스레터 로 영역을 키워갔습니다. 퍼블리셔 회사 입사 후, VueJs 프로젝트 참여하면서, 웹 접근성과 웹 표준을 준수한 마크업 기초를 배우고 컴포넌트(Component) 개발에 관심 두게 됐습니다. 한국 웹 접근성 연구소에서 인증마크 부여된 사이트를 직접 클론 연습하면서, HTML 익숙하도록 노력했습니다.

    • HTML과 JQuery를 통해, 프론트 개발 업무 수행

      한국신용정보원 프로젝트를 통해, 주도적으로 프론트 개발업무를 수행한 적 있습니다. 뷰(View) 관련 업무를 진행하면서, 마크업을 설계하고 스크립트와 플러그인을 사용해, 동적 웹 페이지 개발업무를 성공적으로 마무리했습니다. 코드 유지보수도 고려해야 했고, 업무 중, 백엔드 개발자로부터 JQuery 관련 질문받곤 했습니다. 프론트 업무의 처음과 끝을 경험하면서 스크립트 역량에 관심 두게 됐습니다. 이후 바닐라 자바스크립트를 공부하기 시작했습니다. 함수형 프로그래밍과 RXJS, 타입스크립트 등 자바스크립트 익숙하도록 노력했습니다.

    • "프로젝트 별, 업무 관점 차이를 경험"

      삼성카드 사이트 운영을 맡으면서 프로젝트의 형태가 구축/운영 간, 업무 관점 차이가 존재함을 경험했습니다. 타 직무 이해자(기획자/디자이너/프론트개발자)와 협업하면서 효과적으로 의사소통하고 팀원으로 일하는 방법을 배웠고 고민했습니다. 퍼블리셔와 팀으로 일하는 스크립트 개발자를 지원하면서, 간단한 동적 웹 컴포넌트 스크립트 구현했습니다. 프로젝트 가이드 문서화 중요성과 코드 컨벤션과 같은 협업 규칙에 대해 고민해봤습니다.

    • 프론트 개발자, 다시 기본부터, 새롭게 시작하는 마음으로

      퍼블리셔 직무를 수행하며, 동적 웹 UI 구현 업무를 하고 싶은 욕심이 있었습니다. 프로젝트 협업을 겪으며, 함께 일하는 방법을 몸으로 익혔습니다. 리액트 토이 프로젝트를 만들면서 프로그램 기능의 가짓수나 그 규모가 중요하지 않음을 느꼈습니다. 대신 코드 한줄 한줄이 어떤 의미를 갖는지 좋은 코드 작성하는 목표를 세우게 됐습니다. 현재 그동안 익혔던 퍼블리싱 개념들을 점검하고 있습니다. 가독성 있고 간결하지만, 의미가 분명한 스크립트를 작성하도록, 자바스크립트 엔진 특성을 공부하고 있습니다. 이전 직무를 통해 익힌 지혜와 시간 들을 바탕으로, 팀과 개인 발전에 기여하고 힘쓰는 개발자로 성장하고 싶습니다.

다른 사람과 구별되는 나만의 강점

  • BEM 방법론 적용을 통해, 클래스 컨벤션을 정의 후, 페이지 재작업 시간, 디자인 완료 평균 기간 각각 20분(건당)(30분에서), 2.5일(3일에서)로 단축

    • 좋은 품질의 마크업과 스타일 산출물을 목표로 사이트 운영업무 한 적 있습니다. 마크업 1건을 수정하려고 해당 클래스 명을 검색하면, 같은 스타일의 다른 클래스가 여러 발견하곤 했습니다. 불필요하게 중복된 속성이 검색되어 유지보수가 좋지 않았습니다. 기존 정의된 속성을 가져다 쓰는 운영업무 특성상, 작업자가 중복되게 입력할 여지가 있었습니다. 왜 그럴까 고민해 본 결과, 고유하게 식별되는 클래스 작성하기가 실천되지 않음을 발견했습니다.

    • 이를 해결하기 위해 스타일 클래스 작성 방법론을 검색하였고, 획득한 내용 중에서 유지보수와 쉽게 배워 사용할 수 있는 점에서 BEM방식을 적용하기로 마음먹었습니다. 마크업 구조를 염두에 두며 스타일 정의하는 BEM으로 클래스 컨벤션을 정의하고 사용해, 페이지 재작업 시간을 20분(기존 30분에서)으로, 디자인 완료 평균 기간을, 2.5일(기존 3일에서)로 단축했습니다.

    • 클래스 이름 작성하는 데 시간이 걸린다는 단점을 제외하곤, 스타일 파일만으로 마크업 구조가 예측할 수 있는 점과 이름을 구조적으로 작성할 수 있어, 가독성과 유지보수가 좋았습니다. 다른 작업자로서는, 클래스 명이 고유하여 식별이 빠르고, 수정 부분과 오류 포인트가 바로 보인다는 피드백 받곤 했습니다.

    • 팀 내 클래스 컨벤션이 없어, BEM을 코드 컨벤션으로 요청하려 했지만, 각자 업무수행 방식에 침범할 수 없었고 이슈가 없어, 개인 작업에만 적용해왔습니다.

  • 업무 history 문서화를 통한 의사소통 개선으로, 기획자의 업무 확인 메시지 문의 횟수를 0으로 단축

    • 요약 정리한 작업 history를 사용해, 기획자와 업무 의사소통 이슈를 개선 한 경험이 있습니다. 팀에서 페이지 산출물을 이메일로 전달할 때, 첨부 파일의 폴더구조와 모바일/웹 작업 링크를 함께 적어두는 작업 규칙이 있었습니다. 이것은 작업물에 대해 기획자와 의사소통하는 방식입니다. 기획자 관점에서 정보가 부족했는지 종종 메신저 문의가 오곤 했습니다. 문의 그대로 이메일에 설명글 남겼지만, 도움이 되지 않은 듯 보였습니다.

    • 메신저로 문의받은 질문을 분류했습니다. 대부분 이전 작업 파일 대비 변경 사항은 무엇인지 파일 상태가 작업 중 혹은 최신여부인지 알고 싶어 했습니다. 자주 마크업 업무 요청하는 기획자에게 겪었던 이슈를 나눴습니다. 기획자가 마크업 산출물을 프론트 개발자에게 전달할 때, 개발자가 기획자에게 이메일 포워딩 내용을 퍼블리셔에게 문의한다는 것과 개발자가 어떻게 코드 수정하는지 알게 된 이후, 작업 산출물의 history 정리 습관을 갖게 됐습니다. 어떤 클래스를 변경했는지, 바로 확인할 수 있는 주석명은 무엇인지, 달라진 내용은 무엇인지, 기획자/개발자 관점에서 필요한 정보를 찾을 수 있도록 관련 사항을 기록했습니다. 관련 내용을 이메일에 추가 첨부하여, 기획자/개발자가 산출물을 효과적으로 이해할 수 있었습니다.

기억에 남는 프로젝트

  • 노동요 (팀 프로젝트)

    FE 개발자로 참여해, 3개월가량, 2명 BE 개발자와 1명의 FE와 협업하면서 React에 Typescript를 적용해본 프로젝트
    • 배운 점
      • 간단한 주제라도 이를 실제 서비스로 기능 구현이 중요
      • 사이트 사용자와 제공하는 서비스 개념을 명확히 정의
      • 핵심적인 서비스가 무엇인지, 무엇(what)을 개발해야 할지, 어느 정도(How) 기능이 요구되는지 구체화 과정의 필요
      • 개발 요건을 정함에 있어, 사용하려는 언어의 장점/단점/적응 정도/친숙도 등 논의 후, 선정
      • 팀 회의를 하게 될 경우, 목표가 구체적일 경우 진행하고, 회의를 통해 의미 있는 결과물을 도출하도록 노력
      • 프로젝트 환경셋업과 같은 공통업무를 문서화

  • FleaMarketagora (개인 프로젝트)

    React와 ExpressJs를 활용해 데이터 CRUD를 연습해 본 토이 프로젝트.팀 프로젝트를 수행하며, React 프로젝트를 구현하지 못한 부분을 스스로 테스트하기 위해 진행.
    • 배운 점
      • 코드리뷰의 중요성
      • 코드 품질을 고려하는 태도
      • 컴포넌트 단위로 의미구조를 설계하는 사고방식
      • 사용할 공통 컴포넌트를 정의하고, 핵심 로직 위에 컴포넌트 조립하는 것으로 프로젝트 규모 키우기
      • 렌더링을 최소화하고 메모리를 최적으로 사용하는 코드 작성

개발 프로젝트 진행 시, 주요 담당 역할

프로젝트 코드 활용해, 스크립트 개발자 업무 8건 지원(웹 컴포넌트 스크립트 7건, 롯데월드 카드 소개 슬라이드 스크립트 1건)

  • 삼성카드 운영 프로젝트에서 마크업과 해당 UI 스크립트 업무를 수행한 적이 있습니다. 작성한 스크립트가 프로젝트 제품으로 적합한지 점검받고 싶었고, 이미 다른 곳에서 구현된 사례가 있는지 확인하기 어려웠습니다. 코드 신뢰성을 확보하고 중복을 방지하기 위해, 프로젝트 가이드 문서를 검색했습니다. 스크립트 가이드를 구할 수 있었지만, 최신의 내용을 다루지 않아, 도움 되지 못했습니다.

  • 스스로 참고 자료를 얻기 위해, 사이트의 모든 페이지를 확인했습니다. 조금이라도 유사한 기능이라면, 개발자 도구로 DOM 클래스를 알아내어, 이를 프로젝트 파일에서 찾았습니다. 사이트를 훑어보면서 소스를 확보하는 동시에, 기획자에게 유사한 스크립트 적용된 페이지 있는지 문의했습니다. 직접 코드를 작성하면서, 보완할 점이 없는지 획득한 자료와 비교해보곤 했습니다. 혼자서 끝까지 완성한 이후, 매번 개발자에게 코드 리뷰를 받았습니다. 프로젝트에 사용될 상품이기 때문에, 이와 같은 과정을 반복했습니다. 그 결과 스크립트 8건 업무를 성공적으로 수행할 수 있었습니다. 마지막으로 다른 팀원도 같은 페이지를 수정할 경우를 대비해, 해당 코드를 공유하고 사용법을 알려줬습니다.

  • 업무 수행하며 좋은 코드 작성법을 경험할 수 있었고, 스크립트에 친숙해지는 계기였습니다. 자바스크립트를 제대로 이해하고 싶은 관심과 열정을 갖게 됐습니다.
